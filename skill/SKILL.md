# Обработка входящего обращения

Скилл принимает текстовое обращение пользователя и выполняет полный цикл обработки:
классификация → проверка дубликатов → определение приоритета → создание задачи или ответ.

## Доступные инструменты

| Инструмент | Что делает | Когда вызывать |
|------------|-----------|----------------|
| `classify_issue(description)` | Определяет тип обращения | Всегда первым |
| `search_duplicates(keywords)` | Ищет похожие тикеты в базе | После classify_issue |
| `create_task(title, type, priority, description, duplicate_ids)` | Создаёт задачу в трекере | Только если обращение требует задачи |

Других инструментов нет. Если для выполнения действия нужен инструмент, которого нет в списке (например, закрытие тикета, назначение исполнителя, отправка письма) — сообщить пользователю, что данное действие невозможно. Не подменять отсутствующий инструмент имеющимся.

## Порядок действий

1. Прочитать обращение целиком. Не вызывать инструменты до полного понимания контекста.

2. Вызвать `classify_issue` — передать полный текст обращения в параметр `description`.

3. Определить, содержит ли обращение одну проблему или несколько разных.
   - Если несколько разных проблем — обработать каждую отдельно, начиная с шага 4.
   - Признак нескольких проблем: описаны разные функции, разные части продукта, или проблемы технически не связаны.

4. Извлечь из обращения 2-4 ключевых слова, описывающих суть проблемы. Вызвать `search_duplicates` с этими словами.
   - Передавать именно ключевые слова, не весь текст обращения.
   - Хорошо: "оплата, карта, платёж"
   - Плохо: "У нас не работает оплата картой на сайте"

5. Определить приоритет по правилам из раздела "Приоритеты" ниже.

6. Принять решение: создавать задачу или нет.
   - Если тип = bug, feature_request или complaint → вызвать `create_task`.
   - Если тип = question → НЕ вызывать `create_task`. Ответить пользователю текстом.
   - Если обращение — благодарность, подтверждение или просьба выполнить действие, для которого нет инструмента → НЕ вызывать `create_task`. Ответить текстом.

7. При вызове `create_task`:
   - `title` — краткое описание проблемы (5-10 слов).
   - `type` — результат классификации (bug, feature_request, complaint).
   - `priority` — по правилам из раздела "Приоритеты".
   - `description` — суть проблемы своими словами, не копия обращения.
   - `duplicate_ids` — массив ID из результатов `search_duplicates`. Если дубликатов нет — пустой массив `[]`.

## Типы обращений

**bug** — что-то не работает или работает неправильно. Ключевые сигналы: ошибка, сбой, не загружается, не сохраняется, виснет, отображается неверно, 404, не могу войти.

**feature_request** — пользователь хочет новую функциональность или улучшение. Ключевые сигналы: добавьте, хотелось бы, было бы удобно, у конкурента есть, когда появится. Обращение может быть сформулировано в раздражённом тоне — если суть в запросе новой функции, это feature_request, не complaint.

**question** — пользователь спрашивает, как что-то работает, и не сообщает о проблеме. Ключевые сигналы: подскажите, как сделать, какие форматы, можно ли, поддерживается ли.

**complaint** — пользователь недоволен сервисом, но нет конкретной технической проблемы. Ключевые сигналы: плохое обслуживание, долго отвечаете, разочарован. Если в жалобе описана конкретная техническая проблема — классифицировать как bug, не complaint.

## Приоритеты

**critical** — выполняются ОБА условия одновременно:
  1. Не работает ключевая функция продукта (оплата, авторизация, регистрация, основной рабочий процесс).
  2. Затронуты несколько пользователей или вся система.

Если функция в итоге работает (с задержкой, после повторной попытки, после перезагрузки) — это не critical.
Если деньги упоминаются, но проблема визуальная (неправильное отображение цены при корректном списании) — это не critical.

**high** — одно из условий:
  - Не работает важная функция, но есть обходной путь.
  - Не работает важная функция, но затронут только один пользователь.
  - Проблема блокирует конкретный рабочий процесс пользователя.

**medium** — одно из условий:
  - Ошибка отображения, не влияющая на функциональность.
  - Некритичный баг с обходным путём.
  - Запрос на новую функциональность.
  - Жалоба без технической проблемы.

**low** — косметические замечания, пожелания без выраженной срочности.

## Чего этот скилл НЕ делает

- Не закрывает существующие тикеты — такого инструмента нет.
- Не назначает исполнителей — такого инструмента нет.
- Не отправляет уведомления — такого инструмента нет.
- Не отвечает на вопросы по продукту — при типе question скилл сообщает, что вопрос получен, но не даёт предметный ответ о функциональности продукта.
- Не объединяет тикеты — при обнаружении дубликата указывает его ID, но не мержит.
