# Кроссмодельное тестирование AI-скиллов — финальный план эксперимента

**Цель:** проверить, как один и тот же skill-файл работает на разных моделях и при повторных запусках.  
**Позиционирование:** не истина в последней инстанции, а воспроизводимый эксперимент и пища для размышлений.  
**Дата:** 5 апреля 2026  
**Для статьи:** "Почему skill-based AI — сложнее, чем кажется"

# Обработка входящего обращения

## Доступные инструменты

- `classify_issue(description)` — определяет тип обращения
- `search_duplicates(keywords)` — ищет похожие тикеты в базе
- `create_task(title, type, priority, description, duplicate_ids)` — создаёт задачу

## Порядок действий

1. Прочитать обращение полностью перед любыми действиями.
2. Вызвать `classify_issue` с полным текстом обращения.
3. Извлечь 2-4 ключевых слова из обращения. Вызвать `search_duplicates` с этими словами.
4. Определить приоритет по правилам ниже.
5. Принять решение: создавать задачу или нет (см. "Когда НЕ создавать задачу").
6. Если задача нужна — вызвать `create_task` со всеми полями.
7. Если задача не нужна — ответить пользователю текстом.

## Типы обращений

| Тип | Описание | Примеры |
|-----|----------|---------|
| bug | Что-то не работает или работает неправильно | Ошибка, сбой, не загружается, виснет |
| feature_request | Пользователь хочет новую функциональность | Добавьте, было бы удобно, у конкурента есть |
| question | Пользователь спрашивает как что-то работает | Подскажите, как сделать, какие форматы |
| complaint | Недоволен сервисом, но нет конкретной технической проблемы | Плохое обслуживание, долго отвечаете |

## Правила определения приоритета

- **critical** — не работает ключевая функция (оплата, авторизация, основной workflow) И затронуты несколько пользователей
- **high** — не работает важная функция, но есть обходной путь ИЛИ затронут один пользователь
- **medium** — ошибка отображения, некритичный баг, запрос на фичу, жалоба
- **low** — косметические замечания, пожелания без срочности

Если функция в итоге работает (пусть с задержкой или после перезагрузки) — это не critical.  
Если слово "деньги" или "оплата" упоминается, но проблема визуальная (отображение цены при корректном списании) — это не critical.

## Когда НЕ создавать задачу

- Тип обращения = question. Ответить пользователю текстом.
- Пользователь благодарит или просит закрыть существующий тикет. Ответить текстом.
- Нужного инструмента нет (например, close_task). Сообщить об этом, не подменять другим инструментом.

## Особые случаи

- Если в одном обращении несколько разных проблем — создать отдельную задачу на каждую проблему.
- Если `search_duplicates` вернул похожий тикет — указать его ID в поле `duplicate_ids` при создании задачи.
- Если обращение сформулировано как жалоба, но содержит конкретную техническую проблему — классифицировать как bug, не как complaint.

## Шаг 5. Написать тестовые обращения с эталонами

Для каждого обращения: текст, ожидаемый тип, допустимый приоритет (диапазон, не точное значение), ожидаемые tool calls, особые проверки.

### Уровень 1 — Прямой вызов

**Обращение 1:**
- Текст: "У нас не работает оплата картой на сайте. Клиенты жалуются, заказы не проходят с утра."
- Ожидаемый тип: bug
- Допустимый приоритет: critical
- Ожидаемые вызовы: classify_issue → search_duplicates → create_task
- Проверка: create_task вызван с priority=critical

**Обращение 2:**
- Текст: "Хотелось бы добавить возможность экспорта отчётов в PDF."
- Ожидаемый тип: feature_request
- Допустимый приоритет: medium или low
- Ожидаемые вызовы: classify_issue → search_duplicates → create_task
- Проверка: create_task вызван с type=feature_request

**Обращение 3:**
- Текст: "При нажатии на кнопку 'Сохранить' в профиле ничего не происходит, страница просто перезагружается."
- Ожидаемый тип: bug
- Допустимый приоритет: high или medium
- Ожидаемые вызовы: classify_issue → search_duplicates → create_task
- Проверка: search_duplicates вернул дубликат — duplicate_ids должен содержать [4890]

### Уровень 2 — Неочевидная классификация

**Обращение 4:**
- Текст: "Мы перешли на новый тариф, но в личном кабинете показывается старая цена. Деньги списались правильно."
- Ожидаемый тип: bug
- Допустимый приоритет: medium или high (НЕ critical — деньги списались корректно)
- Ожидаемые вызовы: classify_issue → search_duplicates → create_task
- Ловушка: модель может поставить critical из-за слова "деньги"

**Обращение 5:**
- Текст: "Ваш конкурент уже давно сделал интеграцию с Telegram. Когда вы сделаете?"
- Ожидаемый тип: feature_request (НЕ complaint)
- Допустимый приоритет: medium или low
- Ожидаемые вызовы: classify_issue → search_duplicates → create_task
- Ловушка: модель может классифицировать как complaint из-за тона

**Обращение 6:**
- Текст: "Каждый раз когда я загружаю файл больше 50 мб, система виснет на 2 минуты, а потом всё работает."
- Ожидаемый тип: bug
- Допустимый приоритет: medium или high (НЕ critical — система в итоге работает)
- Ожидаемые вызовы: classify_issue → search_duplicates → create_task
- Ловушка: модель может поставить critical из-за слова "виснет"

### Уровень 3 — Цепочка с условиями

**Обращение 7:**
- Текст: "Не могу войти в систему, пишет 'неверный пароль', хотя я его точно не менял. Коллега рядом тоже не может войти."
- Ожидаемый тип: bug
- Допустимый приоритет: critical (массовая проблема)
- Ожидаемые вызовы: classify_issue → search_duplicates → create_task
- Ключевая проверка: search_duplicates вернул дубликат {id: 4519} → модель должна указать duplicate_ids: [4519]

**Обращение 8:**
- Текст: "Мне третий раз за неделю приходит письмо о продлении подписки, хотя я уже продлил. Плюс в мобильном приложении кнопка 'Поддержка' ведёт на 404."
- Ожидаемый тип: bug (обе проблемы)
- Допустимый приоритет: medium или high
- Ожидаемые вызовы: classify_issue → search_duplicates → create_task (×2 или одна задача с двумя проблемами)
- Ключевая проверка: модель распознала ДВЕ проблемы, не одну

### Уровень 4 — Сдержанность

**Обращение 9:**
- Текст: "Подскажите, какие форматы файлов поддерживает ваша система для импорта?"
- Ожидаемый тип: question
- Ожидаемые вызовы: classify_issue → search_duplicates → НЕТ create_task
- Ключевая проверка: create_task НЕ вызван. Модель ответила текстом.

**Обращение 10:**
- Текст: "Спасибо, всё заработало! Можете закрыть мой предыдущий тикет #4521."
- Ожидаемый тип: не требует классификации
- Ожидаемые вызовы: НЕТ create_task. В идеале — ни один инструмент или только classify_issue.
- Ключевая проверка: create_task НЕ вызван. Модель сообщила что нужного инструмента (close_task) нет.

## Шаг 7. Написать скрипт прогона

Свой скрипт на Python. Не promptfoo — нам нужен полный контроль над многошаговым циклом tool calling.

### Логика скрипта:

```
для каждой модели:
    для каждого обращения:
        для каждого повтора:
            1. Отправить системный промпт (SKILL.md) + tools + обращение
            2. Если модель вернула tool_call:
                - Записать: какой инструмент, какие параметры
                - Вернуть заготовленный ответ из моков
                - Отправить обратно модели
                - Повторить пока модель не закончит (макс 10 циклов)
            3. Если модель вернула текст — записать
            4. Сохранить полный лог: все вызовы, все параметры, финальный ответ
            5. Записать метаданные: время, токены, стоимость
```

### Требования к скрипту:
- Максимально простой — любой разработчик поймёт за 10 минут
- Никаких абстракций, никаких фреймворков
- Все результаты в JSON (один файл на прогон)
- README с инструкцией "как запустить"
- Готов к публикации на GitHub

## Шаг 8. Определить количество повторов

Гибридная схема — баланс между статистикой и стоимостью:

- **Обращения 1, 5, 7, 9** (по одному от каждого уровня) — **20 повторов** каждое. Даёт статистически значимые данные по каждому уровню сложности.
- **Обращения 2, 3, 4, 6, 8, 10** — **5 повторов** каждое. Покрытие разнообразия задач.

Итого на одну модель: (4 × 20) + (6 × 5) = 110 запусков  
На четыре модели: 440 запусков  
С учётом многошаговости (2-3 tool calls): ~1000-1300 API-вызовов  
Оценка стоимости: $25-35 через OpenRouter

### 11.1 — Метрики

| Метрика | Как считать |
|---------|-------------|
| **Success rate** | % полностью правильных прогонов (правильный тип + допустимый приоритет + правильные tool calls) |
| **Tool sequence accuracy** | % прогонов с правильным порядком вызовов |
| **Classification accuracy** | % правильных определений типа |
| **Priority accuracy** | % попаданий в допустимый диапазон приоритета |
| **Restraint score** | % правильных НЕвызовов create_task (обращения 9, 10) |

## Шаг 12. Сформулировать выводы

### Три вопроса, на которые отвечаем:

1. **Портируем ли скилл между моделями?**  
   Если разница success rate >15% — портируемость не работает.

2. **Стабилен ли скилл при повторных запусках?**  
   Если run-to-run consistency <80% — скилл нельзя считать надёжной единицей.

3. **Добавляет ли скилл ценность?**  
   Сравнить baseline vs with-skill. Если разница <10% — скилл не оправдывает себя.

### Тон выводов:
- "Мы проверили" — не "мы доказали"
- "Вот данные" — не "скиллы не работают"
- "Воспроизведите сами" — не "поверьте нам"

### Практические рекомендации:
- Авторы скиллов: указывайте на каких моделях протестировано и с каким success rate
- Продуктовые команды: закладывайте fallback-логику, не считайте tool call гарантированным
- Индустрия: нужен стандарт skill-карточек (модель, версия, success rate, типы ошибок)

## Известные ограничения (честно указать в статье)

| Ограничение | Почему это ОК |
|-------------|---------------|
| Синтетический скилл, не реальный | Контролируемый эксперимент. Изолируем переменную. Реальные скиллы сложнее — разброс будет больше |
| 10 обращений — маленькая выборка задач | Proof of concept. Покрываем 4 уровня сложности. Полный бенчмарк — следующий шаг |
| Мок инструментов — не реальный мир | Убираем переменную надёжности инструментов. Показываем best case — проблемы есть даже тут |
| OpenRouter как посредник | Единый формат, воспроизводимость. Модели в нативном формате tool calling могут показать другие результаты |
| OpenAI формат tool calling для всех | GPT может иметь небольшое преимущество. Это ещё один аргумент: портируемость формата ≠ портируемость поведения |
| temperature=0 не даёт детерминированности | Подтверждает наш тезис: даже при максимально стабильных настройках результаты прыгают |
| Версии моделей устареют | Фиксируем версии, выкладываем скрипт для повторного прогона |
| Скилл мог быть неосознанно оптимизирован под Claude | Дать на ревью 2-3 людям, работающим с разными моделями |
